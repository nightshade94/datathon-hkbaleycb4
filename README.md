# Datathon 2026 Round 1 - The Gridbreaker

This repository contains the reproducible code used for the Datathon 2026 Round 1 submission: MCQ computation, EDA figures, data preparation, and the forecasting pipeline that generates `submission.csv`.

## Repository Structure

```text
data/
  raw/            # Local competition CSV files, not committed
  processed/      # Generated cleaned tables and submission artifacts
images/eda/       # Generated EDA figures used for reporting
knowledge/        # Problem statement and schema notes
notebooks/        # Exploration notebooks
report/           # Runbooks and written summaries
src/              # Reproducible Python pipeline
tasks/todo.md     # Finalization checklist and review log
```

Raw and generated CSV files are intentionally ignored by git. Place the official competition files in `data/raw/` before running the pipeline.

## Setup

Install `uv`, then create the locked environment:

```bash
uv sync
```

The project uses Python 3.10+ and dependencies pinned in `uv.lock`.

## Reproduce Outputs

Run the full local pipeline:

```bash
uv run src/data_prep.py
uv run src/part1_solver.py
uv run src/visualization.py
uv run src/forecasting.py --method auto
```

Main outputs:

```text
data/processed/daily_feature_store.csv
data/processed/baseline_metrics.json
data/processed/submission.csv
images/eda/*.png
```

`submission.csv` is generated with columns exactly `Date,Revenue,COGS` and preserves the row order from `sample_submission.csv`.

## Solution Overview

Part 1 MCQ answers are computed directly from the CSV tables in `src/part1_solver.py`; the script prints the answer choices and can emit supporting evidence with `--json`.

EDA figures are generated by `src/visualization.py` from source tables. The generated plots cover revenue/COGS trend, weekday revenue seasonality, regional revenue, category return rates, traffic-source quality, and missing-value patterns.

Forecasting is implemented in `src/forecasting.py` with leak-safe temporal validation. The `auto` mode compares:

- `linear_regression`: calendar features with weekly/yearly seasonality terms.
- `seasonal_naive`: recursive 7-day lag baseline.
- `seasonal_profile`: historical day-of-year profile scaled by complete-year growth.

The current validation split selects `seasonal_profile`:

| Method | MAE | RMSE | R2 |
|---|---:|---:|---:|
| seasonal_profile | 635,253.88 | 889,452.74 | 0.7159 |
| linear_regression | 1,096,729.73 | 1,453,525.31 | 0.2413 |
| seasonal_naive | 1,722,389.39 | 2,345,896.73 | -0.9762 |

## Reproducibility Notes

- All modeling features are derived from the provided competition files.
- Time-series validation uses the final chronological holdout, not a random split.
- The submission template's `Revenue` and `COGS` placeholders are not used as features.
- Generated artifacts are local and gitignored; rerun the commands above to recreate them.
- There are no committed model weights.
